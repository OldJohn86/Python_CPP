G3 Specs Summary

G3 Documentation Home

Main Processor: 

   Quad-Core Cortex-A53


Peripherals:

DDR : Two (x16) DDR3 chips on board, one of them is reserved.

Flash : 

   one socket for NAND Flash, TSOP48.
   two sockets for serial Flash, SOIC16 and WSON8.

Gigabit Ethernet : 

   4 x RJ45 connectors from four RGMII PHY.
   1 x 1000BASE-X SFP from the RTBI SerDes, which shares with one RGMII port
   4 x RJ45 connectors from QSGMII PHY.

PCIe : 

   two x1 PCIe edge card slots and one x2 edge card slot from each FPGA, where the 2nd lane of the x2 card is through a resistor option from one of the x1 slot. The x2 slot has another resistor option form the RXAUI lanes. Mini-card (WiFi) has to be connected  through an edge card to mini-card adaptor.

RXAUI : 

   one RAXUI daughter card which provides a SFP+ on the daughter card, in ASIC the RXAUI is shared with two PCIe lanes but in FPGA dedicated lanes are used due to FPGA floorplan and timing closure consideration. There is a resistor option to route the RXAUI to the PCIe x2 edge slot as well.

SATA : One SATA connector from each FPGA.

USB : 

   two USB 3.0 standard A connector from each FPGA.
   One USB 2.0 standard A connector from FPGA.

SD / eMMC : 1 x SD card slot, eMMC is through SD to eMMC adapter. 

   SD / eMMC fio performance

SSP : 

   one set of daughter card header to plug in Cortina PCM and audio daughter card, supports PCM TX/RX, I2S TX/RX and S/PDIF TX.

PWR : 

   one power button for power control, one LED to indicate PWR_EN.

UART : two UART pin headers to support two the DB-9 console module.

I2C : one I2C bus to configure the on board devices, such as I2S CODEC, PWM.

SPI : one SPI bus to configure the on board SPI device such as SLIC.

GPIO : one x16 pin header reserved for GPIO test.

RTC : one 32.768KHz oscillator as the RTC reference clock.

G3 Platform SW

What is new in G3 software

ARM Cortex A53 AP and RealTek Torako MIPS PE Toolchains

ARM Cortex A53 Toolchains, Yocto 2.0, OpenWRT and Baremetal

    • GNU GCC 4.8 with glibc-2.19 (for OpenWRT bring)
    • GNU GCC 4.9 with newlib for bare metal, ARM Trusted Firmware and UEFI
  • Previous G2 Bootlooder and POST ported to G3 for FPGA bring up purposes:
    • Uboot (upgraded to newer 2015 U-boot baseline)
    • Cortina G2 Secureboot (first stage) ported to G3
    • Cortina G2 POST (second stage) ported to G3

Realtek MIPS Torako Baremetal Toolchain

To be provided by Realtek for PE development

Next Generation Bootloaders for ARM v8 Secure and Non-secure modes.

ARM Trusted Firmware - Secure Boot Loaders, Secure Runtime Services, and Optional Secure OS

    • ARM's Trusted Firmware BL1 image replaces Cortina Secure Boot (first stage) - region is TZ protected
    • ARM's Trusted Firmware BL2 image replaces Cortina POST (second stage) - region TZ protected
    • ARM's Trusted Firmware BL2 loads BL3-1 image provides ARM SVC Runtime Services in Secure EL3 mode to Non-Trusted Linux - region is TZ protected
    • ARM's Trusted Firmware BL2 loads BL3-2 image (optional) ARM's Trusted Secure OS - TEE. - region is TZ protected
    • ARM's Trusted Firmware BL3 loads BL3-3 image Non-Trusted Bootloader. This will be UEFI. Region is NOT TZ protected.

G3 AP ARM Trusted Firmware build procedure

UEFI (EDK2) Non Secure Loader

    • UEFI runs in ARM Non-Secure mode. It provides UEFI Runtime services to OS's. It can also load EFI binaries which can be loaders.

This is how Linux will be loaded.

UEFI Firmware Interface Support for G3 AP build procedure.

Linux 4.4 Kernel for AP - Non Secure OS

  • Linux-3.18 or newer is current choice which end of Life is Aug, 2016. Will be upgraded to 4.4 to support Yocto 2.1

Flat Device tree for AP

Linux ARMv8 Requires that Board Configuration Information be passed via Flat Device Tree. This is a standard based on OpenFirmware. The goal for this is to allow a common kernel image to be compiled and execute on multiple ARMv8 boards. Only a unique the Flat Device Tree file that is passed in at startup to tell the kernel image which board and configuration it is running on.

G3 SW bring up status

Daily Build status

* 6/14/2016

  1. Continue on G3_FPGA_Bringup_Status

* 8/17

  1. Implement SW workaround to enable direct boot. FPGA ROM copy code from Serial-flash to DRAM and then jump into. But flash interface seems not stable on my FPGA. sometimes cannot identified or fail to erase/program. If programmed correctly, it can auto boot.
  2. SH FPGA can boot to Linux(4 cores) successfully as well.

* 8/12

  1. Slave CPU bring up fail. What I saw is the coherent issue. Slave CPU set online flag in cacheable DDR. But Master CPU still get old value. So it's considered that slave CPU is not up. Master and Slave CPU use same translation table but memory content are different. There is no SCU in ARMv8 CPU. Cache coherent should be handled by cache controller but not functional.

* 8/11

  1. Make tiny ROM code for slave CPUs bring up. Joe is compiling this binary into FPGA bit file. We'll try this tomorrow.

* 8/07

  1. After fixing the SPI tie off (including UART irq) in GIC400 the bare kernel can boot (in failsafe mode), Happy Father's Day!
  2. Will start multi-core bring up next Monday.

* 7/27

  1. CA53 single core with FPU/NEON version works with the uboot, mtest is ok (without $), FPU registers are visible.
  2. GIC400 CPU interface registers can be written correctly after the fabric change from LS to HS1, however the problem now is timer interrupt event is not visible in FPGA-A at all so there is still no interrupt forwarded to the processor, I'm probing FPGA-B (ca53mp4) to make sure interrupt is valid there.
  3. The socket vendor figured out that the V7-585T socket was designed based on a wrong spec. which has the center line deviated by 0.5mm. So we returned the two sockets and they will ship to Japan to fix the problem, expected delivery schedule is 10 days.
  4. [SJC] After fixing the broken wire, the FPGA image loading is fine. Will update the flash with new images and hand over to platform team.

* 7/24

  1. Timer interrupt issue: The interrupt signal pass from CA53 to GIC400, but interrupt signal is not asserted to CA53 CPU interface. We found some GIC registers field are read only. So configuration is not complete. Joe also checked signal and found CFGSDISABLE is configured as High. According to GIC spec :Lockdown is controlled by an active HIGH disable signal, CFGSDISABLE. That is, the system asserts CFGSDISABLE HIGH to disable write access to the register fields and registers. Joe is compiling new FPGA code with CFGSDISABLE tied to low for next try.

* 7/21

  1. The kernel crashes due to illegal instruction. Linux Kernel runs to rest_init() and create kernel threads. So there is regular context switching. FP(Float-Point) and SIMD's register are part of context-switch. This cannot be avoid with compile option because it's implemented in low level ARM64 assemble code. FP registers(q0~q31/fpcr/fpsr) are used in arm64 low level code. We need full ARMv8 CPU logic to bring up Linux kernel.

* 7/16

  1. The D-cache issue in U-boot is clarified. There is a bug in U-boot mmu_setup() which use integer to setup mmu table. This leads to virtual address between 0x80000000~0xFFFFFFFF is not mapped. So that CPU be trapped into exception when access that address range(Peripheral IO). I can run mtest with I$/D$ enabled now, and the speed is faster than before(faster than 10X).

* 7/15

  1. Kernel is unable to boot standalone, CPU is in EL3 in when come from reset. It need some pre init code(U-boot) to migrate it to EL2/EL1. So kernel boot must got through U-boot.
  2. Kernel boot to start_kernel() but dies somewhere in rest_init(). Still tracing...
  3. Serial boot fail: The Identifier debugger shows instruction fetch cycle is not observed by flash controller when CPU come from reset. This transaction is issued by FPGA-B. Seems this transaction got lost somewhere in FA logic. The read cycle be observed on data read or jump PC to flash base with "go 0xe0000000" command.
