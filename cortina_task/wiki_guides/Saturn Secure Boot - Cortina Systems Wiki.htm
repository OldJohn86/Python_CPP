Saturn Secure Boot

Saturn Secure Boot (SSB) is based on ARM Trusted Firmware (ATF) that was original developed for use with the G3 Application Processor (AP). Unlike ATF, the MMU is not configured or used in this context. Instead, Cortina developed Address Translation Unit (ATU) IP unique to Saturn will be used to map high address decoders into the lower 512MB of physical address space. This allows necessary Cortina IP to be access through MIPS KSEG0 (cached)/KSEG1(uncached) logic addressing which only is capable of addressing the lower 512MB of physical address space. In addition, since Saturn does not support a separate CPU context beyond MMU protection, no secure world runtime is supported. Specifically, BL31 and BL32 are not built or loaded by BL2. The only kind of memories protection on Saturn is via MMU. This will be the responsibility of the OS to setup and not SSB. 

The following is summary of images produced from SSB:
  • BL1 - started at the reset vector. Basic HW Init and then copies BL2 into SRAM from non-volatile storage. Can authenticate BL2 using X.509 certificates.
  • BL2 - This image sets up DRAM. It loads BL33 images from non-volatile storage into different memory locations. It also authenticate BL33 using X.509 certificates.
  • BL3-0 - not used
  • BL3-1 - not used due to lack of equivalent secure world CPU mode support.
  • BL3-2 - not used due to lack of equivelent secure world CPU mode support.
  • BL3-3 - U-Boot

Cortina Saturn ATU

Many Cortina IP blocks decode regions are scattered throughout the entire 4GB physical address range of the CPU. However, in order to access physical address above 512MB an MMU must be used in MIPS architecture. Although, Saturn contains an MMU it add significant complexity and overhead to the design to develop, and maintain MMU mapping via the TLB. Therefore, in the context of SSB the use of the MMU will be avoided. Instead, the Cortina Saturn Design has developed an seperate Address Translation Unit (ATU) to remapped physical addressing of Cortina Developed HW components to the lower 512MB of physical address space. The ATU will be programmed with predefined mapping at power-on reset automatically by the Saturn HW.

In order to access Cortina IP HW components below 512MB, a "logical" CPU address alias must be used to specify either cached or uncached access to the physical address. MIPS KSEG0 (cached access) logical space starts at 0x8000_0000 and maps to physical address starting 0x0000_0000. MIPS KSEG1 (uncached access) logical space starts at 0xA000_0000 and maps to the same physical address starting 0x0000_0000. So when the MIPS CPU is being used to access the lower 512MB of address space, the software must either OR in 0x8000_0000 or 0xA000_0000 to the physical address to determine what kind of access to perform. If the physical address is pass to the CPU via software without OR'ing to a KSEG0 or KSEG1 address first then a CPU exception will be thrown as this will attempt to access the MMU which is likely has these addresses unmapped. Passing address like 0x0000_0000 to the CPU will throw as User Space TLB exception because this is mapped to the KUSEG space (MMU controlled). Passing address above 512MB will also cause a TLB exception as those address are MMU controlled for KSEG3. SSB should only use KSEG0 and KSEG1 addresses in software for CPU access.

See section 8.1 of the Realtek RX5281 Datasheet to determine which CPU address map to logical controlled cached access via KSEG0 and KSEG0, vs MMU controlled access via KUSEG and KSEG3.

To determine the ATU mapping in software, a header file MACRO has been developed called ATU_XLAT(x). PRE ATU address will be provided by the Cortina HW team via a registers.h file, or may be looked up by SW engineers during development via the current reg.db file reference above. These address should then be passed through the ATU_XLAT(x) macro. This will yield a physical address below 512MB. Making the address accessible via either KSEG0 or KSEG1. Then this address should be passed through a PHYS_TO_KSEG0(x) or PHYS_TO_KSEG1(x) macro to obtain either a cached or uncached logic address.

The ATU_XLAT macro resides in the SSB code base and should be included in the top level PLATFORM header file include/platform_def.h as #include <ca827x_atu.h>

Build Environment

The Saturn support source code is available via the Cortina git mirrors and is build with the Realtek/LEXRA GCC bare metal toolchain for Taroko family of cores based on MIPS I ISA.

Realtek/LEXRA GCC bare meta toolchain is available via Cortina NFS at the mount point /auto/project2/sw/toolchains/rsdk-4.4.7-5281-EL-rtos-n2.1-m32ut-150702/bin. Please make sure it is in your PATH to access the toolchain. Type: 

 which mips-elf-gcc

confirm that 

/auto/project2/sw/toolchains/rsdk-4.4.7-5281-EL-rtos-n2.1-m32ut-150702/bin/mips-elf-gcc

is returned to make sure the binaries can be found by your development host.

In addition, the following other host tools are required. 

git build-essential uuid-dev iasl bc ncurses-dev device-tree-compiler libssl-dev

Once this is done you can clone the source code with: 

 git clone gitolite@cs-local-gitmirror:sw/platforms/cs-unified/hypervisors/arm-trusted-firmware


Next, cd into the working repository and switch to the Cortina development branch for the Saturn FPGA board 

 cd arm-trusted-firmware
 git checkout cortina/master_v1.3.y
 or
 git checkout cortina/saturn_asic for asic bring up branch

Trusted Board Boot Build (TBB)

Trusted Board Boot uses X.509 certificates to authenticate stages before they are executed. This requires the some additional options in the build, including the use and inclusion of ARM's mbedtls software into the firmware image (formerly known as PolarSSL). In addition, it requires access to private key (Cortina development or customer).

Clone repo which contains Cortina development keys and other useful host tools for development and test. 

cd ..
git clone gitolite@cs-local-gitmirror:sw/platforms/cs-unified/host-tools/ca82xx-devel

Clone mbedtls repo and checkout branch mbedtls-2.2: 

git clone gitolite@cs-local-gitmirror:sw/platforms/cs-unified/packages/oss/mbedtls
cd mbedtls
git checkout mbedtls-2.2
cd ../arm-trusted-firmware

To build BL1 ELF image with full symbols for use with GDB and Saturn QEMU type: 

CROSS_COMPILE=mips-elf- make -f Makefile.mips MBEDTLS_DIR=../mbedtls TRUSTED_BOARD_BOOT=1 \
GENERATE_COT=1  \
ARM_ROTPK_LOCATION=regs \
ROT_KEY=../ca82xx-devel/ATF/TBB/development_keys/saturn_rotprivk_rsa.pem \
V=1 PLAT=saturn_fpga JTAG_DEBUG=1 bl1


To build BL1, and mip image without BL33 (u-boot) simply type: 

CROSS_COMPILE=mips-elf- make -f Makefile.mips MBEDTLS_DIR=../mbedtls TRUSTED_BOARD_BOOT=1 \
GENERATE_COT=1  \
ARM_ROTPK_LOCATION=regs \
ROT_KEY=../ca82xx-devel/ATF/TBB/development_keys/saturn_rotprivk_rsa.pem \
V=1 PLAT=saturn_fpga all fip

For FPGA BUILD:

To build everything with U-BOOT for BL33, then you must first build U-BOOT image. To do this follow the U-BOOT build instructions and then type the following to bundle it into the FIP image: 

CROSS_COMPILE=mips-elf- make -f Makefile.mips MBEDTLS_DIR=../mbedtls TRUSTED_BOARD_BOOT=1 \
GENERATE_COT=1 BL33=../u-boot/u-boot.bin \
ARM_ROTPK_LOCATION=regs \
ROT_KEY=../ca82xx-devel/ATF/TBB/development_keys/saturn_rotprivk_rsa.pem \
V=1 PLAT=saturn_fpga all fip

For ASIC BUILD:

same as above but add in flags for ASIC build 

CROSS_COMPILE=mips-elf- make -f Makefile.mips MBEDTLS_DIR=../mbedtls TRUSTED_BOARD_BOOT=1 \
GENERATE_COT=1 BL33=../u-boot/u-boot.bin TEST_ON_SATURN_ASIC_SIM=1   \
CORTINA_SATURN_ASIC=1 ARM_ROTPK_LOCATION=regs ROT_KEY=../ca82xx-devel/ATF/TBB/development_keys/saturn_rotprivk_rsa.pem \
V=1 PLAT=saturn_fpga CORTINA_SATURN_TESTCODE=1 all fip

How to generate various types of RSA keys and SHA256 hash of public key

Below describes the process that is used to generate private/public keys and ROTPK Hash data found in ca82xx-devel/ATF/TBB/development_keys. A script is provided in ca82xx-devel/ATF/TBB/generate_new_keys.sh which executes the following steps described below:

Generate RSA 2048 bit (PEM) private key (i.e. ROTPK): 

openssl genrsa -out saturn_rotprivk_rsa.pem 2048

Generate RSA 2048 bit public key (DER) from private key (PEM): 

openssl rsa -in saturn_rotprivk_rsa.pem -inform PEM -pubout -outform DER -out saturn_rotpk_rsa.der

SHA256 (binary) hash of public key (DER): 

openssl rsa -pubin -inform DER -in saturn_rotpk_rsa.der -outform der | openssl dgst -sha256 -binary > saturn_rotpk_rsa_sha256.bin

SHA256 (binary) hash of public key(DER) from private key(PEM): 

openssl rsa -in saturn_rotprivk_rsa.pem -pubout -outform DER | openssl dgst -sha256 -binary > saturn_rotpk_rsa_sha256.bin

Saturn QEMU

A QEMU for Saturn may be built by following the instructions here: <Saturn QEMU>

It is recommended that the QEMU working repository be placed at the same level directory level as arm-trusted-firmware and ca82xx-devel directories.

The relative tree structure should look like this: 

arm-trusted-firmware/
ca82xx-devel/
qemu/

This will allow the use of host scripts found in the ca82xx-develop repository to start up both QEMU and GDB with the right options to debug SSB.



For BL1 debug: After a bl1.elf image has been built with debug symbols, per the build instructions above, then follow the instructions below for debug of the image:

Open up a new separate shell window, for the QEMU console output. Execute the following commands to launch QEMU for Saturn: 

$ cd ca82xx-devel/saturn-gdb-qemu-startup
$ ./saturn-qemu-host-start.sh

To quit QEMU the key sequence is Hold down the Control Key and hit "a", then release and hit "x". i.e. Control-a x


Open up another new separate shell window, for interaction with GDB. Execute the following commands to launch GDB and attach to QEMU for Saturn: 

$ cd ca82xx-devel/saturn-gdb-qemu-startup
$ ./saturn-gdb-client-start.sh

To exit GDB type "quit" at the prompt

Note: The original build shell window for ATF, which is a separate shell window from GDB and QEMU, may be maintained to create a full incremental build and test environment.
